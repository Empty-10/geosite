Astova — Technical Validation Brief

GEO / AEO / SEO scanning platform  ·  prepared for independent technical validation  ·  28 June 2026

Astova scans a website the way AI answer engines do and reports, with explicit confidence, what helps or hurts its chances of being retrieved and cited — then hands over the fix. The core promise is determinism: the technical audit is read straight from the live HTML with real parsers and reproduces identically on re-run, so a finding is a verifiable fact, not an LLM's guess.

How to read this document. Section 1 is what it checks today. Section 2 is how (methodology + why it's trustworthy). Section 3 is an honest account of accuracy and limitations. Section 4 is what's missing, and Section 5 is how we plan to close those gaps. The intent is to make the system easy to scrutinise.

The accuracy principle (the design constraint everything obeys)

Every finding carries a confidence label, enforced in the data model:

LabelMeaningExample
VerifiedDeterministic — read directly from the page or an authoritative API. Re-running reproduces it exactly.Title, schema, robots directives, headings, status codes, SSL.
MeasuredSampled from AI engines on a date; always shown with a confidence band + sample size.AI citation / share-of-voice (needs engine API keys).
EstimatedModelled/inferred; flagged as directional.Reserved for inferred signals.

This split is deliberate and is the brand: we never present an estimate as a fact.

1. What Astova checks today

The deterministic engine (Python) runs pure modules; each returns findings with status, severity, evidence (the exact value/line it read) and a plain-English recommendation. ~60 checks across the areas below — all Verified unless noted.

On-page (content & markup)

  • Title tag (presence, length), meta description (presence, length)
  • H1 (missing / multiple / ok), heading hierarchy + heading order
  • Canonical correctness, robots meta (index/noindex), snippet directives (max-image-preview etc.)
  • Open Graph + Twitter card completeness
  • JSON-LD schema: presence, types, required-property validation
  • Image alt coverage + dimensions; internal/external link ratio; anchor-text quality; crawlable anchors; link attributes (rel/noopener); jump links; outbound authority links
  • Language attribute, form labels, hreflang, URL structure

Technical (crawlability & delivery)

  • HTTPS, status codes, redirect chains, HSTS, TLS certificate, mixed content, viewport
  • robots.txt parse — including AI-crawler directives (GPTBot, ClaudeBot, PerplexityBot, Google-Extended)
  • Sitemap validity + freshness; noindex-in-sitemap conflict
  • Response headers: X-Robots-Tag, compression, security headers; resource hints; llms.txt presence

GEO-readiness (the AEO factors — why a page does or doesn't get cited)

  • Up-front answer block (AEO core) — locates the first self-contained answer paragraph and how far down the page it sits
  • Front-loaded opening (a real answer sentence in the first ~150 words), definitive vs hedged language
  • Extractable structure (content lists/tables — navigation menus excluded), summary bullets near the top, chunking (vs walls of text)
  • Quotable-data density (figures, %, currency, dates, measurements), question-style headings, FAQ section (schema or Q&A pairs)
  • Content depth / thin-content, Authority & E-E-A-T signals (author, dates, About/Contact, entity sameAs)
  • JavaScript-dependency — how much content only exists after JS runs (which non-rendering AI crawlers miss)
  • Empty-page collapse — a near-empty (SPA-shell) page reduces to one clear root-cause finding

“What the AI bot saw” (crawler-access reconciliation)

  • Fetches the page as GPTBot and compares with a normal browser fetch — catching WAF/CDN blocks and cloaking that robots.txt parsing alone can't reveal (e.g. Cloudflare's “Block AI bots”). A hard block is treated as critical and leads the verdict.

Local-AEO (conditional — activates only on local-business pages)

  • LocalBusiness schema + subtypes, NAP (name/address/phone) consistency, opening hours, geo coordinates / map embed, Google Business Profile link via sameAs. Skipped entirely on non-local pages so a SaaS page is never penalised.

Performance (authoritative API, on-demand)

  • Core Web Vitals + Lighthouse via Google PageSpeed Insights (LCP, CLS, TBT, FCP, Speed Index; lab + CrUX field). Run on demand because it is slow.

AI visibility Measured (probabilistic, needs engine API keys)

  • Multi-engine prompt sampling across ChatGPT/Claude, Perplexity and Gemini; citation rate, share of voice, cited-source intelligence and sentiment — each shown with a Wilson 95% confidence interval and sample size. Inactive until API keys are configured.

Cross-cutting outputs

  • 20-row scorecard with a single headline “AI Retrievability” score, three critical gates, a hybrid bonus overlay, a plain-English verdict, per-row impact, and a citation-readiness read.
  • Competitor benchmark — your page vs up to 3 rivals, row-by-row leader + where you lead/trail.
  • Crawler-log analytics — verified evidence (uploaded logs or Cloudflare) of which AI crawlers hit which pages, and what errored.
  • Fixes, not findings — deterministic ready-to-apply artifacts (schema, robots.txt, llms.txt, meta, LocalBusiness) plus AI-drafted rewrites for judgment-dependent issues.
  • MCP server — exposes the engine to AI assistants (ChatGPT / Claude / Cursor): audit_url, scan_url, fix_plan (agent-actionable remediation) and audit_project (pre-deploy audit of a local project + optional localhost crawl, source never leaving the machine).

2. How it works (and why it's trustworthy)

Deterministic, reproducible engine

Modules are pure functions: parsed HTML in, a list of findings out. No module reaches into another, and parsing works on an HTML string, so the whole engine is offline-testable. The same input always yields the same output — re-running a scan reproduces every number. This is the key difference from “paste your HTML into an LLM and ask for a score”: there is no sampling variance.

Rendering

Pages are fetched raw and, when they look like a client-rendered shell, rendered headlessly (Cloudflare Browser Rendering) so the engine scans what a JS-executing crawler sees — while still reporting how much content was JS-dependent (what non-rendering AI crawlers miss).

Scoring

Findings aggregate into a 20-row scorecard mapped to a recognised GEO/AEO/SEO audit structure. Each row scores from the status of its checks; three critical gates (no up-front answer → 0; promotional intro → capped; navigation-only “summary” → capped) prevent a page from scoring well while failing the things that actually govern citation. A small all-or-nothing bonus overlay rewards combinations (schema breadth, root files, delivery). The headline is explicitly a readiness composite, labelled as such.

Independent validation built in

An accuracy benchmark harness compares Astova's deterministic checks against Google Lighthouse on the same pages. It has already caught and fixed real bugs (e.g. an alt-text false positive, a link-text rule that didn't scale). The technical/on-page reads match Lighthouse on the hard, unambiguous signals. The test suite is 216 automated tests, all green; linting is clean.

3. Accuracy & honest limitations

For a validator, the important nuance is the difference between measurement correctness and predictive validity:

  • Measurement correctness — strong. Every deterministic check reads the live HTML and reproduces exactly. This part is verifiable and solid.
  • Predictive validity — directional, not yet proven. The GEO-readiness checks are grounded in 2026 GEO research, but the headline score has not been calibrated against real citation outcomes. It is presented as a readiness proxy, not a measured probability — by design.
Strong checksHeuristic checks (known error margin)
Up-front answer block; quotable-data density; bot-access; JS-dependency; schema/FAQ; HTTPS/SSL/status; robots/sitemap.E-E-A-T (partly CSS-class inference); promotional-intro wordlist; hedged-language ratio; the 220/350-word answer thresholds.

Other current limitations: AI-visibility sampling needs engine API keys to run and currently covers three engines; persistence/shareable links require the production database path to be set; the public remote MCP endpoint is unauthenticated; the engine itself lacks an SSRF guard (the web route has one); and the web layer has no automated tests yet (the engine has 216).

5. How we plan to close the gaps

Sequencing

  1. Business layer first. Accounts + Stripe billing + multi-tenant, so the tool can be sold. Set the production database path so persistence and shareable reports go live.
  2. Monitoring + alerts. An external scheduler triggers a “run-due” endpoint that re-scans tracked URLs, diffs against the last scan (the diff engine already exists), and emails regressions — flagship alert being AI-crawler access lost. Deterministic and near-zero cost; gated as the paid recurring feature.
  3. Hardening. SSRF guard on the engine, a token on the remote MCP, and a web-layer test suite — required before promoting the public surfaces.
  4. Closed loop. AI-referral attribution (GA4 + log join), then deeper citation sampling (more engines) once keys/budget are in place.
  5. Distribution. Publish the MCP (npm/pip) and a WordPress plugin once the brand/name is finalised; an AEO content/guides layer that also dogfoods the product.

Cost posture: the deterministic engine makes no paid API calls, so scanning, the competitor benchmark, and developer/localhost audits are effectively free to run; the only metered costs are optional headless rendering and the (opt-in) AI-visibility sampling. For developers, the local MCP runs the engine on their own machine, so that compute is theirs and their source never leaves it.
